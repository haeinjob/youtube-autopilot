## 1. 왜 만들려고 했나 (목적)

주언규 PD의 40주 유튜브 수익화 커리큘럼을 학습하면서, 매번 Claude에게 직접 물어보는 대신 **"이미 학습이 완료된 AI를 웹에서 바로 쓸 수 있으면 좋겠다"**는 생각에서 출발.

목표:
- Claude Code에서 나(AI)와 대화하는 것처럼, 주언규 강의 내용을 완전히 체화한 AI와 웹사이트에서 대화
- 아이템 추천, 수익화 전략, 콘텐츠 기획 등 강의 내용 기반 즉답
- 친구한테 카톡하듯이 자연스럽고 실용적인 답변

## 2. 무엇을 만들었나 (구현 현황)

### 기술 스택
- **Frontend**: Next.js 14 (App Router), TypeScript, CSS
- **AI Engine**: 최종 → Google Gemini 1.5 Flash API
- **배포**: Vercel (main 브랜치 연동)
- **저장소**: github.com/haeinjob/youtube-autopilot

### 파일 구조
```
webapp/
├── app/
│   ├── api/chat/route.ts   ← AI 응답 API (핵심 파일)

│   └── layout.tsx
├── data/
│   └── knowledge_base.md  ← 주언규 강의 요약 지식베이스 (24,206자)
```

- 주언규 강의 1~13주차 핵심 내용 요약
- 24,206자 분량
- 포함 내용: 결 이론, 역산 공식, 객단가 이론, 키/풀링 콘텐츠, 제목 디벨롭 8단계, 썸네일 6단계, 1군 선수 기준, 12주차(키쇼츠·쿠팡파트너스), 13주차(키콘텐츠기초2편) 등

## 3. 겪은 문제들과 해결 과정

### 문제 1: Vercel Production Branch 불일치
- 증상: 배포는 되는데 환경변수가 안 먹힘
- 원인: Vercel이 `main` 브랜치를 Production으로 보는데 코드가 `master`에 있었음
- 해결: `master`에서 `main` 브랜치 생성 후 push → Vercel이 `main`을 추적하게 됨

### 문제 2: Groq API 토큰 한도 초과 (핵심 문제)
- 증상: `413 Request too large` 오류
- 원인: 지식베이스(24,206자 ≈ 17,000 토큰) + 시스템 프롬프트 합산이 Groq 무료 티어 TPM 한도 초과
  - `llama-3.3-70b-versatile`: TPM 12,000 → 요청 17,173 토큰 → 초과
  - `llama-3.1-8b-instant`: TPM 6,000 → 더 심각
- 해결 시도:
  1. 지식베이스 13,000자로 자름 → 여전히 초과
  2. RAG 구현 (질문 관련 섹션만 포함) → 어느정도 해결했으나 광범위한 질문에 한계
  3. **최종: Google Gemini 1.5 Flash로 전환** (TPM 1,000,000 → 문제 해결)

### 문제 3: AI 응답이 끊김
- 증상: 답변이 중간에 잘림 (예: "다른 어떤 콘텐츠보다도 팬" 에서 종료)
- 원인: Vercel 서버리스 함수 기본 타임아웃 10초. Gemini가 긴 응답 생성 중 10초 초과 → 강제 종료
- 해결: `export const maxDuration = 60;` 추가 → 60초로 연장
- **그러나**: 여전히 응답 시작까지 1분 이상 걸리는 문제 잔존

### 문제 4: AI 답변 품질 (일반론적인 답변)
- 증상: "CTR을 높이세요", "주언규 PD님의 강의에 따르면~" 같은 교과서적 답변
- 원인: 시스템 프롬프트가 지식베이스를 "참고 자료"로만 활용하도록 설계됨
- 해결 시도:
  1. 말투 변경 (카카오톡 대화체)
  2. 프레임워크를 시스템 프롬프트에 직접 임베드 (역산 공식, 결 이론, 객단가 기준 등)
  3. 질문 유형별 강제 변환 규칙 추가 (조회수 질문 → 제목 디벨롭 8단계 / 아이템 질문 → PASS/FAIL 판정)
  4. 구조화된 진단 리포트 형식 (⚡판정 / 💰수익설계 / 🎯결분석 / 📹콘텐츠구조 / ✅즉시실행)
- **아직 미완**: 1분 넘게 걸리다 보니 실제 품질 개선 여부를 제대로 테스트 못 함

## 4. 왜 1차 중단했나

**핵심 이유: 응답 속도 1분 이상**

Gemini 1.5 Flash가 24,000자짜리 지식베이스 + 긴 시스템 프롬프트를 처리하는 데 시간이 너무 걸림. 실용적인 도구로 쓰기 어려운 수준.

추가 이유:
- 응답 품질을 제대로 검증하기 전에 속도 문제가 선행
- 추가 개선에 시간 투자 대비 즉각적인 효용이 낮음
- Claude Code에서 직접 대화하는 게 현재로서는 더 효율적

## 6. 재개할 때 해결해야 할 것들

### 우선순위 높음
1. **응답 속도 문제** - 가장 큰 문제. 선택지:
   - **옵션 A**: 지식베이스를 더 압축해서 토큰 수 줄이기 (현재 24,206자 → 10,000자 이하 목표)
   - **옵션 B**: Gemini API를 Edge Runtime에서 실행 (fs 대신 import로 KB 로드)
   - **옵션 C**: 유료 AI API (Claude API, GPT-4o) - 속도 훨씬 빠름
   - **옵션 D**: Groq Dev Tier 결제 ($10~20/월) - 토큰 한도 없음 + 초고속

2. **AI 답변 품질 검증** - 속도 해결 후 실제로 테스트 필요

### 우선순위 낮음
3. UI 개선 (현재 기본 채팅창, 토스 스타일 대시보드로 업그레이드 가능)
4. Supabase 연동 (대화 기록 저장 등)
5. 시장성 검증 PASS/FAIL 대시보드 화면

## 7. 지식베이스 현재 내용 요약

`webapp/data/knowledge_base.md`에 포함된 주요 내용:
- 핵심 수익화 공식 (역산 알고리즘)
- 결(맥락) 이론
- 상품 선정 기준 (객단가 25만원 법칙, 1군 선수 기준)
- 나로부터 출발하는 아이템 시스템
- 키 콘텐츠 / 풀링 콘텐츠 구조 (4:1 비율)
- 3주차: 키 콘텐츠 기획
- 4주차: 풀링 콘텐츠 기획
- 5주차: 제목 디벨롭 8단계
- 6주차: 썸네일 이미지 디벨롭 6단계
- 7주차: 원고 작성 기초 (ChatGPT 활용)
- 8주차: 원고 작성 심화 (공감/정보/행동 버튼)
- 11주차: 풀링 콘텐츠 레벨테스트
- 12주차: 키쇼츠 → 쿠팡파트너스 전략 (진입 7조건, 마진율 구조 등)
- 13주차: 키콘텐츠 기초 2편 (쇼츠 판매 5조건, LTV 개념, 협업 단가 협상법)

**미포함 (아직 강의 수강 안 함)**: 14~40주차
